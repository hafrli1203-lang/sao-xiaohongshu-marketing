# 에이전트 봇 개발 플로우

이 프로젝트는 아래 5단계 플로우를 반드시 순서대로 따른다.
단계를 건너뛰거나 병합하지 말 것.

---

## STEP 1. 설계 문서

코드 작성 전 아래 항목을 모두 문서로 정의한다.

- **목적**: 이 봇이 해결하는 문제 한 줄 정의
- **트리거 조건**: 언제 실행되는가 (시간, 이벤트, 수동 등)
- **실패 복구**: API 오류, 타임아웃, 예외 발생 시 어떻게 처리하는가
- **상태 경계**: 타 봇·모듈과 공유하는 상태가 있는가, 충돌 가능성은 없는가
- **아웃풋 정의**: 최종 결과물의 형태·포맷·전달 경로

> 설계 문서가 불완전하면 다음 단계로 넘어가지 않는다.

---

## STEP 2. 스펙 테스트

설계 문서 기준으로 정상 케이스를 먼저 검증한다.

- 설계에 명시된 트리거로 실행했을 때 의도한 흐름대로 동작하는가
- 아웃풋이 STEP 1에서 정의한 형태로 나오는가
- 보수적/수동 방법 금지 — 기존 인프라 활용 최단경로로 구현

---

## STEP 3. 아웃풋 체크

결과물 품질을 확인한다.

- 포맷·구조가 정의와 일치하는가
- 전달 경로(Telegram, Sheets, Notion 등)가 정상 작동하는가
- 사람이 읽었을 때 즉시 판단 가능한 형태인가

---

## STEP 4. 인풋 체크

예외 상황을 검증한다.

- 엣지케이스 입력 시 어떻게 반응하는가
- STEP 1에서 정의한 실패 복구 로직이 실제로 동작하는가
- 빈 값, 형식 오류, API 무응답 케이스 포함

---

## STEP 5. 모듈화 명령

검증 완료된 기능을 분리한다.

- 기능 단위로 독립 모듈 분리
- 타 봇과의 인터페이스 명확히 정의 (입력/출력 형식, 호출 방식)
- 재사용 가능한 구조로 정리

---

## 공통 원칙

- 이해 문서(설계) 없이 코드 먼저 짜지 않는다
- 보수적·수동적 방법을 디폴트로 제안하지 않는다
- 기존 인프라(GAS, Telegram 봇, Claude API, Notion)에 통합 가능한 형태로 구현한다
- 버전은 소수점으로 관리한다 (v1.0 → v1.1 → v2.0)
- 버그 발견 시 번호 부여 후 버전 올려서 수정 기록한다
